# Summarizer

## 프로젝트 개요

카카오톡 비즈니스 채팅 메시지를 표준화된 요약 형식으로 변환하는 **Windows Forms 유틸리티 앱**이다.
성형외과 클리닉 직원이 고객과의 상담 예약 메시지를 정해진 양식으로 빠르게 정리하기 위해 사용한다.

- **플랫폼**
  - 지금은 .NET 8 / Windows Forms
  - 향후 버전에서는 변경할 수 있음
- **빌드 결과물**: WinExe (단일 실행 파일)
- **현재 버전**: 1.1.2 (`Program.cs`의 `Version` 상수 및 `.csproj`의 `FileVersion`/`AssemblyVersion` 동기화 필요)

## 작업 방식에 대한 일러두기

- CLAUDE.md 및 작업 지시에 관련한 .md 파일들은 .claude 디렉토리의 하위에 위치해라. 
  - 작업 계획 및 지시에 대한 .md 파일들은 여러 개일 수 있다.
  - 필요에 따라, 하위 디렉토리 구조를 더 깊게 하여 .md 파일들을 배치할 수 있다.
- 나의 지시사항을 수행한 결과에 대해, **네가 무엇을 수행했는지에 대한 상세한 보고서를 작성해라.**
  - 일반적으로 .md 형식으로 작성해라.
  - 더 효과적인 보고서 형식이 있다고 판단했다면, 이를 나에게 추천하고, *내가 승인한 뒤*, 그 형식으로 보고서를 작성할 수 있다.
- 작업 계획 디렉토리의 구조
    ```
    Summarizer/
    ├── .claude                    # Claude에 관한 모든 계획 및 작업 지시 파일들은 이 디렉토리에 넣는다.
        ├── CLAUDE.md              # 이 문서가 위치해야 할 지점
        ├── settings.json          # 기존의 ../Summarizer/.claude에 있던 내용은 여기로 이동
        ├── settings.local.json    # 기존의 ../Summarizer/.claude에 있던 내용은 여기로 이동
        ├── hooks                  # 기존의 ../Summarizer/.claude에 있던 내용은 여기로 이동
        └── plans                  # 모든 작업 계획 및 지시서들은 이 디렉토리 안에, 계획한 버전 별 디렉토리에 위치한다.
            ├── v1.1.2             # 버전 1.1.2에 대한 작업 계획 및 지시서는 모두 여기에 위치한다.
                ├── research.md    # 기존 구현의 분석을 비롯하여, 해당 버전의 목표 기능에 대한 분석 내용을 여기에 작성해라. (버전 디렉토리 별로 존재)
                ├── plan.md        # research.md의  내용을 토대로 하여, 이 버전에서 수행할 실제 구현 계획을 여기에 작성해라. (버전 디렉토리 별로 존재)
                ├── task.md        # plan.md에서 도출한 구현 계획에 따른 세부 작업 항목 별 수행 결과를 여기에 작성해라. (버전 디렉토리 별로 존재)
            ├── v1.2.0             # 버전 1.2.0에 대한 작업 계획 및 지시서는 모두 여기에 위치한다.
            └── vX.X.X             # 버전 X.X.X에 대한 작업 계획 및 지시서는 모두 여기에 위치한다.
    ```
  - 작업 디렉토리의 범위는 특별한 지시가 없는 한, Summarizer/ 디렉토리 내에서만 수행한다.
    - 그 외의 위치에 작업이 필요한 경우에는, 내 승인을 받아야 한다.
- 각 버전 별 작업 계획 및 지시서의 일반적인 지침
  - research.md
    - 내가 지시한 버전 또는 마일스톤 목표의 기능들을 깊이 있게 이해하고, 가능한 상세히 파악하여 그 결과 보고를 research.md 파일에 작성하라.
    - 해당 버전 또는 마일스톤의 research.md를 완료하기 전에는 프로젝트에 대해 어떠한 구현 및 변경도 하지 말 것.
    - 내가 research.md의 완료를 승인한 뒤에 plan.md를 만들어라.
  - plan.md
    - research.md의 보고 내용을 토대로, 구현 계획을 가능한 상세하게 작성하라.
    - 반드시 research.md를 완료하고, 이를 내가 승인한 이후에 작성해야 한다.
    - 내가 plan.md의 완료를 승인한 뒤에 task.md를 만들어라.
    - 작성하는 내용은 각 스토리 단위로 식별 가능해야 하고, 진행과 완료 여부를 다음과 같은 tag로 구분할 수 있어야 한다.
      - [신규] (또는 [ToDo])
      - [진행] (또는 [InProgress])
      - [완료] (또는 [Complete])
  - task.md
    - task.md에는 plan.md의 내용을 모두 구현하기 위한 실제 과업 목록들을 만들고, 각 과업마다 수행 결과를 작성하라.
    - 반드시 plan.md를 완료하고, 이를 내가 승인한 이후에 작성해야 한다.
    - 작성하는 내용은 각 과업 단위로 식별 가능해야 하고, 진행과 완료 여부를 다음과 같은 tag로 구분할 수 있어야 한다.
      - [신규] (또는 [ToDo])
      - [진행] (또는 [InProgress])
      - [완료] (또는 [Complete])
  - 지시 대상을 파악할 때 건너뛸 수 있는 항목들
    - [완료] (또는 [Complete]), [닫음] (또는 [Closed]) tag가 붙은 계획 및 과업은 지시사항을 파악할 때 건너뛸 수 있다.
    - 수정 또는 재작업이 필요한 경우, 내가 [완료]가 아닌, 다른 tag로 변경할 것이다.
  - 그 외, 프로젝트 진행에 더 효과적이거나 일반적으로 권장하는 방안에 대해 추천할 수 있으며, 내 승인을 받고 적용할 수 있다.

## 코딩 규칙

- 프로젝트에서 이미 사용 중인 .editorconfig가 있다면, 이를 사용할 것.

- 기존 .editorconfig가 없으면 새로 .editorconfig를 만든다. 이 때 다음과 같은 규칙을 기본적으로 포함하여 생성하라.
  - 소스파일 인코딩은 UTF-8 (No BOM) 사용
  - 중괄호 스타일은 언어 문법 상의 문제가 없다면, 항상 BSD 스타일로만 사용할 것.
    - 단, 한 줄로 간단히 표현 가능한 경우에는 {} 기호를 모두 한 줄에 합쳐 쓸 수 있다.
    - 분기 및 반복문의 본문을 한 줄로 간단히 표현 가능한 경우에는 {} 기호를 생략할 수 있다. (단, 언어 문법에서 허용할 수 있는 경우에 한함)
  - 탭 사용 금지. 소스코드의 구조 공백은 Space 4칸 사용
  - 각 줄의 마지막 공백들은 모두 제거
  - 줄바꿈은 파일마다 단일하게 사용 (CR / LF 또는 LF). 한 파일 내에서 섞어 쓰지 말 것.
  - 필드, 매개변수는 camelCase 사용
  - 그 외 타입(네임스페이스 / 클래스 / 구조체 / 인터페이스) 이름, 함수(Method), 속성(Property)는 PascalCase 사용
  - 전처리기는 대문자와 '_'로만 사용
  - switch 구문에서 case 구문을 switch보다 들여쓰지 말 것.

- 이름 짓는 방식
  - 내가 특별히 지시하지 않는다면 축약어를 사용하지 말 것.
  - 반복문 변수는 기본적으로 'index'와 같이, 최소 3글자 이상인 단어 사용. (i, j, k 식으로 단문자 사용하지 말 것)
  - Template, Generic 등으로 타입이 이름이 매우 길어지는 경우, 이를 typedef 또는 using 등을 사용하여 축약할 수 있다.

- 코드 블록의 세부 규칙
  - (대개 분기문 및 getter, setter) 코드에서는 중괄호가 대칭적이어야 한다.
    - 선호하는 스타일
      ```csharp
      string Text
      {
          get
          {
              return text;  
          }
          set
          {
              text = value;
              CheckValidationText(value);
          }
      }

      if ()
      {
          // Do something 1st
          // Do something 2nd
      }
      else
      {
          // Do something single line
      }
      ```

    - 선호하지 않는 스타일
      ```csharp
      string Text
      {
          get => text;
          set
          {
              text = value;
              CheckValidationText(value);
          }
      }

      if ()
      {
          // Do something 1st
          // Do something 2nd
      }
      else
          // Do something single line
      ```

## 저장소 사용 규칙

- git은 commit까지는 자동으로 수행해도 좋다.
- 다만, git push는 내 승인을 받고 진행해야 한다.
- git commit에는 반드시 수행한 내용에 대한 comment를 작성해야 한다.
  - 관련이 있는 task.md의 과업 제목
  - 수행한 내용을 요약하여 작성
- git comment는 한국어와 영어로 작성할 수 있다.

## 개인정보 및 테스트 데이터 보호 규칙

- 이 프로젝트는 실제 고객의 상담 대화, 이름, 연락처 등 **개인정보가 포함될 수 있는 데이터를 처리**한다.
- 실제 대화 내용이나 개인정보가 담긴 파일(예: `requirements.txt`)은 **절대로 저장소에 커밋해서는 안 된다.**
  - 해당 파일들은 `.gitignore`로 추적에서 제외한다.
- 기능 테스트를 위한 예제 시나리오가 필요한 경우, **반드시 허구의 인물과 가공의 내용**으로 작성한다.
  - 실제 이름, 실제 전화번호, 실제 날짜 조합 등 식별 가능한 정보를 포함해서는 안 된다.
